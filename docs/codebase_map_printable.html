Executive Summary

Citizen Photometry is a Windows-first PySide6 desktop application for astrophotography analysis. Architecturally, it is best understood as a mode-driven desktop shell wrapped around a large domain-heavy core package. It is not a thin GUI layered on top of a small backend; most of the repository's complexity sits in astronomy, imaging, catalog, and workflow logic.

The most important structural fact is that photometry_app/ui/main_window.py is the orchestration hub. It wires together menus, workflow state, rendering surfaces, and feature entry points for the application's seven major modes: Differential Photometry, HR Diagram, Asteroid/Comet Detection, Astrostack, Transient Finder, Sky View, and Sky Explorer.

The photometry_app/core/ package contains most of the system's behavior: file scanning, WCS validation and plate solving integration, catalog lookup, photometric measurement, differential matching, light-curve generation, report export, solar-system queries, moving-object discovery, transient search, synthetic tracking, calibration, alignment, and HR-diagram analysis.

Main structural takeaway: treat the codebase as a desktop application shell with three primary layers: UI surfaces, async orchestration, and astronomy-processing engines.

The main separation between interactive UI and expensive work is photometry_app/ui/workers.py, which acts as the background execution boundary for long-running CPU-heavy and network-heavy tasks.

From a maintenance perspective, the repository is a broad but understandable monolith: there are meaningful subsystem boundaries, but the top-level shell imports deeply across them, so coupling is highest at the application entry surface.

Shared contracts and configuration sit in photometry_app/core/models.py and photometry_app/core/settings.py, making those files essential for anyone trying to understand how data and settings flow through the system.

Subsystem Summary and Reading Order

1. Entry and Shell

The runtime starts in photometry_app/main.py, which configures Qt application attributes, performs startup cleanup, and launches MainWindow. The shell then centralizes mode selection, menu actions, theme state, and feature entry points in photometry_app/ui/main_window.py.

2. UI Surface

The photometry_app/ui/ package contains dialogs, annotation/image panels, light-curve and HR widgets, labeling flows, level controls, and specialized sky-view rendering files. The UI layer is broad because multiple astronomy workflows live in one desktop frame.

3. Async Boundary

photometry_app/ui/workers.py is the main keep-the-GUI-responsive layer. It coordinates background tasks and bridges the UI to heavy computation, catalog lookups, and export work.

4. Shared Contracts

photometry_app/core/models.py defines the shared domain vocabulary: modes, WCS state, scan results, photometry measurements, and other cross-cutting data carriers. core/settings.py persists global and workflow-specific settings.

5. Photometry Backbone

The base image-analysis path is spread across scanner.py, pipeline.py, photometry.py, matching.py, plotting.py, and exporters.py. Together these files implement scan, solve/validate, measure, compare, summarize, and export.

6. Astronomy Services and Specialized Engines

Catalog and field intelligence live primarily in catalogs.py, wcs.py, solar_system.py, and sky_explorer.py. Advanced workflows build on top of that foundation in discovery.py, transient.py, synthetic_tracking.py, hr_diagram.py, alignment.py, and calibration.py.
